SphereTest Project Explanation (Detailed)
Date: 2026-04-02 | Last Updated: ✅ Hotfix #6 COMPLETE - Late Join Session Transition

============================================================
1) HIGH-LEVEL OVERVIEW
============================================================
SphereTest is a full-stack web project with:
- backend: Node.js + Express + MongoDB (Mongoose) + Socket.io + JWT auth + express-validator
- frontend: React (Vite) + Tailwind + Framer Motion + axios + socket.io-client

Main idea:
- Admin/teacher creates a "Sphere" (a test room with settings).
- Students join by game code.
- Questions can be MCQ, CODE, TEXT, BOOL.
- Backend exposes REST APIs with validation and authorization.
- Frontend provides interactive UI and connects end-to-end to backend.
- Real-time features via Socket.io: session starts, answers submitted, leaderboard updates.

Core user flow:
1. Admin registers
2. Admin creates sphere, sets title/type/duration, gets game code
3. Admin shares code to students
4. Students register and join sphere via code
5. Admin starts session (Socket.io event)
6. Students see questions, submit answers in real-time
7. Leaderboard updates as answers arrive
8. Admin ends session, final rankings displayed

2) BACKEND ARCHITECTURE
============================================================
Location: backend/src

Entry and startup:
- server.js:
  - Loads dotenv
  - VALIDATES JWT_SECRET exists in process.env; fails fast if missing (P0 security fix)
  - Connects MongoDB via config/db.js
  - Creates HTTP server from Express app
  - Attaches Socket.io with CORS
  - Registers enhanced socket handlers with real-time events
  - Starts listening on PORT

Express app:
- app.js:
  - express.json middleware
  - CORS using CLIENT_URL env (default localhost:5173)
  - health route: GET /
  - mounts APIs:
    - /api/auth (register, login, me)
    - /api/users (protected with auth + role checks)
    - /api/spheres (create, list, find by code, join)
    - /api/questions (create, list, update, delete)
  - global notFound and errorHandler middleware

Middleware:
- middleware/auth.js:
  - protect: verify JWT token, populate req.user
  - roleCheck(role): verify user.role matches (admin/student)
  - No hardcoded fallback secrets (P0 fix)

- middleware/validation.js (NEW):
  - validateRegister: name, email, password (min 6), role enum
  - validateLogin: email, password
  - validateCreateSphere: title, type enum, duration number
  - validateCreateQuestion: questionText, type enum
  - handleValidationErrors: catch and return 400 with error array

Database:
- config/db.js:
  - Mongoose connection using MONGO_URI
  - fallback local URI: mongodb://localhost:27017/spheretest

Models:
- User: name, email, password (hashed), phone, role (admin/student), createdAt
- Sphere: title, description, type, createdBy (ref), participants (array of user IDs),
  gameCode (6-char alphanumeric), startTime (optional, for scheduled sessions),
  endTime (optional, explicit session end time; overrides startTime + duration if set),
  duration (default 60 mins, used to calculate endTime if endTime not explicitly set),
  maxPlayers, difficulty, security (faceId, fullscreen, tabSwitchDetection),
  sessionStatus (DRAFT|UPCOMING|ACTIVE|ENDED), actualStartTime (Date)
  * Session timing: If endTime exists, use it. Otherwise compute as startTime + duration
  * Auto-start (NEW): Sessions auto-start at startTime; admin can override to start early
  * actualStartTime: Records when session truly started (prevents double-starts)
- Question: sphereId, type (MCQ/CODE/TEXT/BOOL), questionText, options (for MCQ),
  correctAnswer, codeLanguage (for CODE), starterCode, createdAt

Socket Events (NEW in Part 3, Enhanced Hotfix #5):
- join_sphere: client joins room for real-time updates
- start_session: starts test (auto-triggered at startTime OR manual admin click)
  * Backend records actualStartTime, updates sessionStatus to ACTIVE
  * Broadcasts session_started with timing info to all in room
- submit_answer: student submits answer, stored in-memory, broadcasts answer_received
- progress_update: emits answered count to admin
- session_end: admin ends test, broadcasts session_ended + final rankings
  * Backend updates sessionStatus to ENDED
- leaderboard_update: emitted after each answer, shows current rankings

Frontend: SphereLobbyPage now includes:
- Countdown timer for scheduled sessions (shows "Auto-starting in X seconds")
- Auto-trigger start_session when startTime reached
- Admin can manually click button to start anytime
- On-demand sessions skip countdown, require manual start

Routes:
- authRoutes: register, login, me (all with validation)
- userRoutes: GET (protected+auth), POST (protected+admin role)
- sphereRoutes: create, list, get, get by code, join, delete (with validation)
- questionRoutes: create, list, update, delete (with validation + ownership check)

4) REAL-TIME SOCKET.IO EVENTS (Part 3)
============================================================

Server-side: sockets/socketHandler.js

Session Lifecycle:
- join_sphere: Student connects with { sphereId, userId }. Added to room.
- start_session: Admin emits → server broadcasts session_started { sphereId, startTime } to room.
  Students' test UI activates on receipt.
- submit_answer: Student emits { sphereId, questionId, answer, userId } → server stores in memory Map
  indexed by sphereId/questionId/userId. Emits answer_received to admin. Emits progress_update to admin
  with { userId, answered, total }.
- leaderboard_update: After each answer_received, server recomputes scoring (1 point per correct),
  sorts participants, broadcasts leaderboard { rankings: [{userId, name, score}] }.
- session_end: Admin emits → server broadcasts session_ended + final leaderboard to room.

Client-side: services/socket.js

Connection:
- connectSocket(token): Connects to server with auth bearer token.
- disconnectSocket(): Teardown.

Emitters:
- emitStartSession(sphereId): Triggers session start.
- emitSubmitAnswer(data): Submits {sphereId, questionId, answer, userId}.
- emitSessionEnd(): Ends session.

Listeners:
- onLeaderboardUpdate(cb), onSessionStarted(cb), onSessionEnded(cb), onAnswerReceived(cb), onProgressUpdate(cb):
  Register callbacks for real-time updates.
- removeSocketListener: Unregister listeners.

============================================================
Session Lifecycle:
1. Admin clicks "Start Session" → emitStartSession(sphereId)
2. Server broadcasts session_started to all students in room
3. Students see questions, submit answers → emitSubmitAnswer(data)
4. Server stores in-memory Map<sphereId, Map<userId, answer>>
5. Server broadcasts answer_received + updates leaderboard to all
6. Admin clicks "End Session" → emitSessionEnd(sphereId)
7. Server computes rankings, broadcasts session_ended + final leaderboard

Memory Storage:
- In-memory Map cleared per session (no persistence yet)
- Rankings computed by answer timestamp (first correct wins priority)
