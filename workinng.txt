BACKEND (fully functional)
1. Server startup pipeline:
   - env load
   - JWT_SECRET validation (fails fast if missing)
   - DB connect
   - express app + socket server creation

2. Authentication APIs:
   - POST /api/auth/register (with validation: name, email, password min 6, role enum)
   - POST /api/auth/login (with validation: email, password)
   - GET /api/auth/me (token required)
   - All routes use bcrypt for password hashing

3. User Management APIs:
   - GET /api/users (token required + auth middleware)
   - POST /api/users (token + admin role required)

4. Sphere APIs:
   - POST /api/spheres (token required, with validation, accepts optional endTime field)
   - GET /api/spheres
   - GET /api/spheres/:id
   - GET /api/spheres/code/:code
   - POST /api/spheres/join (token required, validates session timing: rejects join if session ended)
     * Session timing: Uses explicit endTime if provided, falls back to startTime + duration
     * On-demand spheres (no startTime): always joinable until full
     * Scheduled spheres (startTime provided): joinable before endTime (can pre-join)
     * After endTime: join rejected with "Session has ended"
   - DELETE /api/spheres/:id (creator only, 403 if not owner)
   - Session state now computed consistently across frontend and backend

5. Question APIs:
   - POST /api/questions (token required, with validation)
   - GET /api/questions/:sphereId
   - PUT /api/questions/:id (token required, creator only, with validation)
   - DELETE /api/questions/:id (token required, creator only)

6. Mongo schemas enforce base constraints and relationships

7. Socket.io real-time events with JWT authentication:
   - Server: Middleware verifies JWT token on handshake, rejects unauthenticated connections
   - Attaches socket.userId and socket.user to authenticated sockets
   - Events: join_sphere, start_session, submit_answer, progress_update, session_end, leaderboard_update, answer_received
   - Answer scoring: MCQ/BOOL binary, TEXT/CODE with partial credit
   - Only authenticated users can emit events

8. Seed script (seedData.js) initializes sample data

9. Input validation layer:
   - register: name (required, min 2), email (valid), password (min 6), role (enum)
   - login: email (valid), password (required)
   - create sphere: title (required), type (enum), duration (number, min 1)
   - create question: questionText (required), type (enum), sphereId (required)
   - All return 400 with descriptive errors on validation failure
